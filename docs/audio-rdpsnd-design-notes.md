# RDP Audio Playback — design notes (pre-implementation)

Status: **implemented** (rdpsnd-over-DVC in webrdp-min, NativeMedia/audio_ss4s on the C
side). These are the findings and the chosen design for adding audio output (remote PC
sound → TV speakers) to gnomecast-native, kept as the design record.
Goal restated: the TV plays music/sounds generated by the remote GNOME PC, with hardware
decoding on the TV where possible.

Everything below was verified against the actual sources (gnome-remote-desktop master on
GitLab, the vendored IronRDP fork, the vendored ss4s tree, and our own code) on 2026-07-02.

---

## 1. What the server (gnome-remote-desktop) actually does

Source: `src/grd-rdp-dvc-audio-playback.c` in gnome-remote-desktop (uses FreeRDP's rdpsnd
*server* implementation, `<freerdp/server/rdpsnd.h>`).

- **Transport: DYNAMIC virtual channel, not the static `rdpsnd` channel.** grd registers
  audio playback as a DVC (file literally named `grd-rdp-dvc-audio-playback.c`; the DVC
  name per MS-RDPEA / FreeRDP is `AUDIO_PLAYBACK_DVC`). A client that only implements the
  static rdpsnd channel will never receive audio from grd.
- **Formats advertised** (all stereo, `N_CHANNELS = 2`):
  | Tag | Value | Rate | Notes |
  |---|---|---|---|
  | `WAVE_FORMAT_AAC_MS` | 0xA106 | 44100 | avgBytes 12000, blockAlign 4, bits 16 |
  | `WAVE_FORMAT_OPUS` | 0x704F | 48000 | avgBytes 12000, blockAlign 4, bits 16 |
  | `WAVE_FORMAT_PCM` | 0x0001 | 44100 | 16-bit, blockAlign 4, avgBytes = rate×4 |
- **Codec preference when the client accepts several: AAC > Opus > PCM** (explicit
  if/else-if chain over `aac_client_format_idx` / `opus_client_format_idx` /
  `pcm_client_format_idx`). If the client accepts none of the three → "Terminating
  protocol".
- **Wave transport: `SendSamples2` → SNDC_WAVE2 PDUs.** grd checks
  `rdpsnd_context->clientVersion < 8`, so the client must advertise **protocol version 8**.
- **Handshake required before any audio flows**: server sends Server Audio Formats →
  client must reply Client Audio Formats; server sends Training → client must reply
  Training Confirm within `PROTOCOL_TIMEOUT_MS = 10s`, else teardown.
- **QoS**: server tracks render latency (via wave confirms) and drops frames when it
  exceeds `MAX_RENDER_LATENCY_MS = 300`. So the client should send WaveConfirm promptly.
- Packetization: PCM at `PCM_FRAMES_PER_PACKET = 1024` frames; Opus/AAC frame counts come
  from `grd_rdp_dsp_get_frames_per_packet()`. Each `SendSamples2` call carries one encoded
  buffer (for Opus: one Opus packet per Wave2 — exactly what NDL wants, see §3).

## 2. What the vendored IronRDP gives us

Crate `third_party/IronRDP/crates/ironrdp-rdpsnd` (already in the tree, NOT yet a
dependency of webrdp-min):

- `client.rs` implements the rdpsnd client **as a static virtual channel only**
  (`SvcProcessor`, channel `b"rdpsnd\0\0"`). There is **no DVC variant** anywhere in the
  tree (`AUDIO_PLAYBACK_DVC` string absent). → the static client is useless against grd
  as-is, BUT:
- **The `pdu` module is public and complete**: `AudioFormat { format: WaveFormat,
  n_channels, n_samples_per_sec, n_avg_bytes_per_sec, n_block_align, bits_per_sample,
  data: Option<Vec<u8>> }` (derives Eq/Hash); `WaveFormat::{PCM=0x0001, OPUS=0x704F,
  AAC_MS=0xA106, ...}`; `ServerAudioOutputPdu` (AudioFormat/Training/Wave2/Volume/Pitch/
  Close variants); `ClientAudioOutputPdu`; `Version::V8`; `QualityMode::High`;
  `AudioFormatFlags::ALIVE`. So we can build our own DVC processor out of these without
  touching the fork.
- The reference client state machine to mirror (`client.rs`): on ServerAudioFormats →
  reply `client_formats()` (client's supported ∩ server's advertised, via HashSet
  equality; volume 0xFFFF, pitch 0x00010000, dgram_port 0, flags ALIVE) and, if server
  version ≥ V6, a `QualityMode::High` PDU; on Training → TrainingConfirm; on Wave2 →
  `handler.wave(format_no, ts, data)` then WaveConfirm(ts, block_no). Note: the static
  client only handles Wave2 (v8+); the v<8 Wave path is a TODO — irrelevant for us since
  grd is v8.
- `ironrdp_dvc::DvcProcessor` trait (`crates/ironrdp-dvc/src/lib.rs:36-37,45-56`):
  `channel_name()/start()/process()/close()` returning `Vec<DvcMessage>`, where
  `pub trait DvcEncode: Encode + Send {}` and `pub type DvcMessage = Box<dyn DvcEncode>`.
  We already attach one DVC this way (EGFX `GraphicsPipelineClient` inside
  `DrdynvcClient::with_dynamic_channel`, `webrdp-min/src/native.rs:636-642`).
- **Protocol-level gotcha**: the connector Config currently sets
  `enable_audio_playback: false` (`webrdp-min/src/native.rs:621`), which makes IronRDP set
  the `NO_AUDIO_PLAYBACK` flag in the Client Info PDU — the server then won't stream audio
  at all. Must become `true`.

Our existing Rust↔C plumbing (all confirmed in code):

- `RdpCallbacks` `#[repr(C)]` struct: Rust `webrdp-min/src/native.rs:76-87`, C mirror
  `native/include/rdp_ffi.h:36-44`. Layout is tested on BOTH sides:
  `native.rs` test `ffi_struct_layout_is_c_compatible` (~line 1451) and
  `native/tests/test_rdp_ffi_layout.c` (CTest `rdp-ffi-layout`).
- All callbacks fire synchronously on the `"rdp-worker"` thread (spawned in
  `rdp_session_start`, `native.rs:1352-1354`); buffers are valid only during the call.
- `CallbackSink` (`native.rs:107-181`) is Copy+Send — a copy can be handed to a DVC
  handler for direct synchronous callback dispatch (the EGFX handler queues into an
  `Arc<Mutex>` drained by `drain_gfx`, but audio doesn't need reordering/PTS synthesis,
  so direct calls are simpler; to be sanity-checked at implementation time).

## 3. What ss4s / the TV can play

Vendored tree `third_party/ss4s`. The module our app uses on the TV is **ndl-webos5**.

- **ndl-webos5 audio codecs: `SS4S_AUDIO_PCM_S16LE` and `SS4S_AUDIO_OPUS` ONLY.**
  (`modules/webos/ndl/webos5/ndl_audio.c`: GetCapabilities = PCM|OPUS, maxChannels 6;
  OpenAudio's switch has no AAC/AC3 branch.) This is an NDL DirectMedia v2 API limit
  (`NDL_AUDIO_TYPE` = PCM/MP3/OPUS only).
- **Opus is decoded by the TV hardware/system** — ss4s passes encoded packets straight
  through (`NDL_DirectAudioPlay`). The in-tree libopus targets are shims only:
  `ss4s-opus-empty` = precomputed silent packet to prime the decoder after load;
  `ss4s-opus-fix` = decode/re-encode remap only for the 5.1 channel-mapping corner case
  (stereo never touches it). **Feed = one Opus packet per `SS4S_PlayerAudioFeed` call**;
  `codecData` (OpusHead) optional for stereo.
- PCM path: interleaved S16LE, sample rate must be one of the NDL discrete set (48000,
  44100, 32000, ... — both our candidates are fine). `samplesPerFrame` in `SS4S_AudioInfo`
  is **ignored** by ndl-webos5/smp (only software mixers use it) — set something sane
  (960 for Opus @48k = 20 ms, 1024 for PCM to match grd) and move on.
- AAC would only be possible via the `smp-webos` (Starfish) module (accepts ADTS AAC-LC),
  but mixing SMP-audio with NDL-video means two separate player contexts fighting over
  the single hardware media resource — untested, risky. Not worth it when Opus exists.
- **Audio and video share ONE `SS4S_Player`** (and on NDL one underlying media pipeline).
  Opening either track triggers a full `NDL_DirectMediaUnload` + `Load` (both tracks
  reload). `SS4S_PlayerSetWaitAudioVideoReady(true)` would defer the load until BOTH
  tracks are open — but then a lone track never loads (breaks video-only and audio-only
  sessions), so we keep `wait=false` and accept one reload glitch when the second track
  opens. In practice rdpsnd format negotiation completes before the first video AU (EGFX
  needs caps+surface round-trips first), so the expected order is: audio opens →
  audio-only load; first IDR arrives → video opens → one reload (audio hiccup of a few
  hundred ms, video starts clean). Acceptable.
- **Thread safety**: `SS4S_PlayerAudioFeed` has its own feed-guard (refcount + condvar on
  close) and the NDL module serializes audio feed against open/close with an internal
  mutex. Feeding audio and video from the same rdp-worker thread is safe; so is a
  separate thread later.
- **The blocker in our current code**: `native/src/video_ss4s.c` embeds the entire SS4S
  lifecycle inside `NativeVideo` — `SS4S_Init` with **`.audioDriver = NULL`** (line 101,
  audio driver never initialized!), `SS4S_PlayerOpen` in `native_video_init_ss4s`
  (87-147), `SS4S_PlayerClose` + `SS4S_Quit` in `native_video_close` (267-284). So today:
  (a) no audio driver exists at all, (b) audio couldn't survive the video close/reopen
  that happens on desktop-size change (`main.c` `on_video_au`), and (c) in the
  RemoteFX/RGBA graphics path no `NativeVideo` (hence no player) exists at all.

## 4. Chosen design

### Codec strategy
Advertise **Opus (48000/2ch) + PCM (44100/2ch/16-bit)**, deliberately NOT AAC.
grd's preference order (AAC > Opus > PCM) then selects **Opus** → encoded packets go
straight to NDL → **hardware decode on the TV** (the user's "по возможности аппаратное
декодирование"). PCM stays as the universal fallback (needs no decode at all).
To dodge `AudioFormat` Eq/Hash mismatches on aux fields (avgBytes/blockAlign), build the
client format list by **filtering the server's own advertised structs** (echo back the
server's exact entries whose tag/rate/channels we support) instead of hardcoding.

### Rust side (`webrdp-min`, no fork patches)
- Add path dep `ironrdp-rdpsnd` (pdu module only).
- Flip `enable_audio_playback: true` (`native.rs:621`).
- New `RdpsndDvcHandler` implementing `ironrdp_dvc::DvcProcessor` for
  `"AUDIO_PLAYBACK_DVC"`, registered as a second `.with_dynamic_channel(...)` on the
  existing `DrdynvcClient` (next to EGFX, `native.rs:636-642`). State machine mirrors
  `ironrdp-rdpsnd/src/client.rs`: ClientFormats(V8, ALIVE, volume 0xFFFF) + QualityMode
  (High) → TrainingConfirm → per-Wave2: fire C callback, reply WaveConfirm(ts, block_no)
  promptly (grd's 300 ms latency QoS). Volume/Pitch/Close: log/ignore (TV volume is
  controlled by the TV remote).
- Holds a `CallbackSink` copy; calls C callbacks directly (synchronously on rdp-worker).
- **ABI (append-only) — two new `RdpCallbacks` fields** + C enum `RdpAudioCodec`
  (1=OPUS, 2=PCM_S16LE):
  - `on_audio_format(ctx, uint32_t codec, uint32_t sample_rate, uint16_t channels)` —
    fired on negotiation and on any format change (Wave2's format_no maps to a different
    codec/rate/channels than last fired).
  - `on_audio_data(ctx, const uint8_t *data, size_t len, uint32_t ts_ms)` — one encoded
    Opus packet (or PCM chunk) per call; bytes valid only during the call.
- Extend `empty()`, `CallbackSink` wrappers, both layout tests, and add unit tests that
  drive the DVC state machine with encoded ServerAudioOutputPdu bytes and assert the
  ClientFormats/TrainingConfirm/WaveConfirm replies + callback capture.

### C side — the `NativeMedia` refactor
- New `native/src/media_ss4s.c` + `native/include/media_ss4s.h`: owns what
  `native_video_init_ss4s` owns today — module selection (same candidate list, but
  `SS4S_Config.audioDriver = videoDriver = module->id`), `SS4S_Init`/`SS4S_PostInit`/
  `SS4S_PlayerOpen`/`SetWaitAudioVideoReady(false)`/viewport, teardown
  `SS4S_PlayerClose`+`SS4S_Quit`. Exposes the player handle, viewport setter, and
  capability queries.
- `video_ss4s.c` slims down: `native_video_open(NativeMedia*, w, h, fps)` only attaches
  the video track (`SS4S_PlayerVideoOpen` + the H264/size capability check moves here);
  `native_video_close` only `SS4S_PlayerVideoClose`. Annex-B/keyframe logic untouched.
- New `native/src/audio_ss4s.c` + header: `native_audio_open(NativeMedia*, codec, rate,
  channels)` → `SS4S_PlayerAudioOpen`; `native_audio_feed` (NOT_READY/OVERFLOW →
  drop+log-once, ERROR → error); `native_audio_close`.
- `main.c`: `App` gains `NativeMedia *media; NativeAudio *audio;` under the existing
  `video_lock` (same thread pattern as video). `on_audio_format`: close old audio, ensure
  media, open audio — **failure degrades to silent video, never session-fatal**.
  `on_audio_data`: feed; on feed error log once + close audio (degrade).
  `native_stop_rdp`: close audio → video → media. The RGBA-switch path in
  `on_bitmap_update` must close only the video track, keeping media+audio alive.
  Viewport updates move from `native_video_set_viewport` to the media object.
  Non-SS4S builds: all audio calls are no-op stubs (same pattern as video).
- `rdp_ffi.h`: append the two fields + enum; extend `test_rdp_ffi_layout.c`;
  `native/CMakeLists.txt`: add the two new sources to the executable list.

### Failure/degradation policy
Audio is best-effort throughout: any negotiation, open, or feed failure logs
`[native-audio] ...` and leaves a working silent-video session. No new terminal states.

## 4b. Findings from the live bring-up (things the research above missed)

Three additional blockers surfaced on the real TV↔grd pair, all fixed:

1. **grd disables audio for clients without network autodetection.** In
   `rdp_peer_post_connect()`: *"Client does not support autodetecting network
   characteristics. Disabling audio output redirection"* when the client lacks
   `RNS_UD_CS_SUPPORT_NETCHAR_AUTODETECT`. Advertising the flag alone is dangerous:
   FreeRDP sends autodetect PDUs over the **MCS message channel** with no
   `messageChannelId == 0` guard (they'd go to channel 0 and kill our session), and once
   the flag is advertised the FreeRDP server **blocks in connect-time autodetect** waiting
   for RTT responses before licensing. So the fork now negotiates the message channel
   (GCC `ClientMessageChannelData` → `ConnectionResult.message_channel_id`), answers
   connect-time RTT/BW requests in the connector's `ConnectTimeAutoDetection` state, and
   answers continuous RTT pings in the session's x224 processor (grd uses those to
   estimate audio render latency).
2. **TrainingConfirm `wPackSize` must echo the wire value.** grd sends Training with
   `wPackSize=1024` and validates the confirm against exactly 1024, silently ignoring
   mismatches until its 10 s protocol timeout tears the DVC down. IronRDP's decoder
   subtracts the 8 prolog/header bytes when sizing `data`, so the confirm computes
   `data.len() + 8` (risk R4 from this doc, confirmed live).
3. **`SS4S_GetAudioCapabilities()` always fails on ndl-webos5.** The parameterless
   variant passes `SS4S_AUDIO_NONE` as the wanted-codec mask and the module returns
   false on an empty match. `native_audio_open` must use
   `SS4S_GetAudioCapabilitiesByCodecs(&caps, codec)`.

4. **"Track-only" close does not exist on the webOS backends** (review finding). Both
   `SS4S_PlayerAudioClose` (ndl `CloseAudio` → `UnloadMedia`) and `SS4S_PlayerVideoClose`
   (ndl `CloseVideo` → `UnloadMedia`; smp `StarfishPlayerUnloadInner`) unload the WHOLE
   shared pipeline without reloading the surviving track. Consequences and fixes:
   a mid-session audio feed error must **mute** (`native_audio_disable`) instead of
   closing the track (closing would freeze video in eternal `FEED_NOT_READY`); the
   RGBA-switch path reopens the audio track after closing video (cheap — Opus/PCM need no
   keyframe); an audio format change under live video closes the video track so
   `on_video_au` reopens it on the next SPS+PPS+IDR rather than feeding P-frames into a
   freshly reset hardware decoder.

5. **Audio dropouts under heavy video traffic → client jitter buffer.** Audio and video
   share one TCP connection; bursts of large 4K frames delay the audio DVC packets, which
   then arrive in a late burst (arrival gaps of 60–230 ms measured in live logs) while
   the NDL buffer runs dry — audible sub-second dropouts. Server-side QoS was ruled out:
   grd estimates render latency purely from the client's WaveConfirm `wTimeStamp` (grd
   sends its waves with timestamp 0, our confirms echo it, and grd skips zero values —
   its 300 ms drop logic never fires for us). Fix: `native_audio_feed` fronts a
   configurable jitter prebuffer (`audioPrebufferMs`, default 60, 0 disables) — it
   accumulates that much audio before the first hardware feed and re-arms after source
   pauses, so playback always carries a safety margin at the cost of the same added audio
   latency; arrival gaps above 60 ms are logged as evidence for future tuning.

Also observed (benign): one `omxopusdec ... failed to setup output format` GStreamer
error on the audio-only NDL load, gone after the full audio+video reload; noisy
`gst_object_unref` GStreamer-CRITICAL warnings from the platform pipeline during reloads.

## 5. Open items to verify at implementation time (small, code-level)

1. The concrete pattern for returning our PDUs as `DvcMessage`: check how ironrdp-egfx
   wraps its client→server messages as `DvcEncode` (blanket impl vs explicit) and copy it
   for `ClientAudioOutputPdu` (worst case: local newtype implementing Encode by
   delegation + `impl DvcEncode`).
2. Exact field names of `ClientAudioFormatPdu` / `TrainingConfirmPdu` / Wave2 variant in
   `ironrdp_rdpsnd::pdu` (mirror `client_formats()` exactly).
3. Confirm `DrdynvcClient::with_dynamic_channel` chaining for a second channel and DVC
   name matching semantics.
4. Enumerate every `main.c` call site of `native_video_open/close/set_viewport/width/
   height` for the minimal-diff refactor; check whether any CTest target links
   `video_ss4s.c` (then `media_ss4s.c` must be added there too).
5. Whether a direct C-callback call from inside `DvcProcessor::process` violates any
   locking invariant (expected no: the C handlers only fprintf + call SS4S and never call
   back into Rust — same contract as `on_video_au`).
